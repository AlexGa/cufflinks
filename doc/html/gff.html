GFF files


Some of the Cufflinks modules take as input a file (or more) containing known gene annotations or other transcript data in GFF format (General Feature Format). GFF has many versions, but the two most popular that are supported by Cufflinks (and other programs in the Tuxedo suite, like Tophat) are GTF2 (Gene Transfer Format, described here) and GFF3 (defined here). Here are a few notes about the way these formats are interpreted by the Cufflinks programs.

GTF2

According to the GTF2 description, the transcript_id attribute is required by our GFF parser, and a gene_id attribute, though not strictly required in our programs, is very useful for grouping alternative transcripts under a gene/locus identifier. An optional gene_name attribute, if found, will be taken and shown as  a symbolic gene name or short-form abbreviation (e.g. gene symbols from HGNC or Entrez Gene). Some annotation sources (e.g. Ensembl) place a "human readable" gene name/symbol in the gene_name attribute, like a HUGO symbol (while gene_id might be an automatically generated, numeric identifier for the gene).

TopHat and Cufflinks generally expect exon features to define a transcript structure, with optional CDS feature to specify the coding segments. Our GFF reader will ignore redundant features like start_codon, stop_codon when whole CDS features were provided, or *UTR features when whole exon features were also given. However, it is still possible to provide only CDS and *UTR features and our GFF parser will reassemble the exonic structure.

GFF3

As defined by the GFF3 specification , the parent features (usually transcripts, i.e. "mRNA" features) are required to have an ID attribute, but here again an optional gene_name attribute can be used to specify a common gene name abbreviation. If gene_name is not given, it can be also inferred from the Name or ID attributes of the parent feature of the current parent mRNA feature (if given in the input file). Exon or CDS features are required to have a Parent attribute whose value must match the value of the ID attribute of a parent transcript feature (usually a "mRNA" feature).

Feature restrictions

For various reasons, we currently assume some maximum values for the length (genomic span) of various gene and transcript features provided in the GFF:
  • genes cannot be longer than 7 Megabases
  • exons cannot be longer than 30 Kilobases
  • introns cannot be larger than 6 Megabases
Also, transcripts IDs are expected to be unique across all the provided GFF records (though we relaxed this restriction by limiting it to a chromosome/contig scope).

Due to these requirements, Cufflinks programs may fail to load the user provided GFF file, and an error message should specify the offending GFF record. The user is expected to remove or correct such GFF records in order to continue the analysis.

The gffread utility


A program called gffread is included with the Cufflinks package and it can be used to verify or perform various operations on GFF files. For example, in order to make sure that a GFF file from a certain annotation source is compatible with our GFF reader, gffread can be used to load the transcripts from the file and print these transcripts back, in either GFF3 (default) or GTF2 format, keeping only the essential attributes, so the user can verify if the transcripts in that file were "understood" correctly. The command line for such a visual inspection of a given GFF file could be:

gffread annotation.gff -o- | more

This will show the minimalist GFF3 re-formatting of the input file (annotation.gff in this example). In order to see the GTF2 version of the same transcripts, the -T option should be specified, like this:

gffread annotation.gff -T -o- | more

gffread can also be used to perform other useful tasks, like generating a FASTA file with the DNA sequence of all transcripts found in the input GFF records (given the genomic sequence also). This could be accomplished with a command line like this:

gffread -w transcripts.fa -g hg19.fa annotation.gff
