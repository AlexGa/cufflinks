GFF (General Feature Format)


Some of the Cufflinks modules take as input a file (or more) containing known gene annotations or other transcript data in GFF format (General Feature Format). GFF has many versions, but the two most popular that are supported by Cufflinks (and other programs in the Tuxedo suite, like Tophat) are GTF2 (Gene Transfer Format, described here) and GFF3 (as defined here).

GTF2

The transcript_id attribute is the only one required by the GFF input of most Cufflinks programs, but a gene_id attribute, though not strictly required in our programs, is very useful for grouping alternative transcripts under a gene/locus identifier. An optional gene_name attribute, if found, will be taken and shown as  a symbolic gene name or short-form abbreviation (e.g. gene symbols from HGNC or Entrez Gene).
Some annotation sources (e.g. Ensembl) place a "human readable" gene name in that attribute, like a HUGO symbol (while gene_id might be an automatically generated, numeric identifier for the gene).

TopHat and Cufflinks generally ignore redundant features like start_codon, stop_codon when whole "CDS" features were provided, or UTR features when whole "exon" features were also given.

GFF3

As defined by the GFF3 specification , the parent features (usually transcripts, i.e. "mRNA" features) are required to have an ID attribute, but here again an optional gene_name attribute can be used to specify a common gene name abbreviation. If gene_name is not given, it can be also inferred from the Name or ID attributes of the parent feature of the current parent mRNA feature (if given in the input file). Exon or CDS features are only required to have a Parent attribute, according to the specification.
